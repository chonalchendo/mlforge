# v1.0.0 - Production Ready

**Target Release:** When feature-complete and stable (estimated Q2 2026)
**Status:** Planned
**Breaking Changes:** None after v1.0.0 (stable APIs)

## Overview

v1.0.0 marks mlforge as **production ready** - a complete feature store SDK suitable for real-world ML workflows. This release establishes stable APIs with backwards compatibility guarantees.

This document defines the **definition of done** for v1.0.0.

---

## Core Value Propositions

A production-ready feature store must deliver:

1. **Online/Offline Alignment** - Same features for training and serving, preventing training-serving skew
2. **Point-in-Time Correctness** - Temporal joins that prevent data leakage
3. **Multiple Compute Backends** - Local development (Polars, DuckDB) and production scale (PySpark)
4. **Multiple Storage Backends** - Cloud-agnostic (S3, GCS, Unity Catalog) and multiple online stores (Redis, DynamoDB, Databricks)
5. **Data Quality** - Validators, quality metrics, and drift detection
6. **Developer Experience** - Clean APIs, CLI tools, and programmatic discovery

## Success Criteria

v1.0.0 is successful when:

1. **Users can define features** with clean, intuitive APIs
2. **Features work at scale** from local development to production datasets
3. **Online/offline alignment** prevents training-serving skew
4. **Multiple cloud providers** are supported (AWS, GCP, Databricks)
5. **Data quality is monitored** with validators and drift detection
6. **Documentation enables self-service** without extensive support
7. **APIs are stable** and backwards compatible

---

## Design Principles

Throughout v1.0.0 development, adhere to:

1. **Simple interfaces** - Common cases require minimal code
2. **Deep modules** - Hide complexity inside, not in the API
3. **Sensible defaults** - Works out of the box, customizable when needed
4. **Fail fast with helpful errors** - Clear messages with hints
5. **Consistency** - Same patterns across engines and stores
6. **Extensibility** - Easy to add new engines/stores
